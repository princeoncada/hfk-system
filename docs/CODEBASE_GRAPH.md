# HFK Publishing Engine - Graphify Workflow Reference

## Purpose

CODEBASE_GRAPH.md is the authoritative Graphify workflow reference for this repository. Codex and Claude Code should read this file when they need to understand which graph artifact to use, when to refresh the graph, or which CLI commands to run.

## Three Artifacts and When to Use Each

| Artifact | What it is | When to read | Why | Limitation |
| --- | --- | --- | --- | --- |
| `codebase-graph.json` | Compact normalized JSON generated by `generate-codebase-graph.ps1` from Graphify output. | Every Codex prompt that touches app code or scripts. | Gives the file list, symbol map, imports, and route surfaces in a format compact enough for most prompts. | Compact; not a substitute for source reads on complex orientation questions. |
| `graphify-out/GRAPH_REPORT.md` | Graphify's native narrative report: communities, god nodes (most-connected abstractions), and extraction stats. | Orientation-heavy work: entering an unfamiliar area of the codebase, planning a multi-file feature, or debugging a cross-cutting concern. | Provides richer relational context, including which symbols are central and which files cluster together, that `codebase-graph.json` compresses away. | Larger file; not needed for routine single-file edits. |
| Direct source file reads | The current implementation files selected after graph orientation. | Always required before editing a file. | Graph artifacts describe structure; source reads reveal current implementation details. | Source reads are targeted by the graph, but they still consume more context than the compact artifact. |

## Before / After - What Changed with Graphify

| Before (5.1.x and earlier) | After (5.2.x+) |
| --- | --- |
| Codex read 5-8 source files to orient itself at session start. | Codex reads `STATE.json` + `codebase-graph.json` first (2 files). |
| Each prompt opened multiple files to answer "which file handles X?" | `codebase-graph.json` answers "which file handles X?" for most cases. |
| No compact map; file selection was driven by filename guessing. | Orientation-heavy prompts add `GRAPH_REPORT.md` (3 files total). |
| Source reads often started broad. | Source reads are narrowed to 2-3 targeted files instead of 5-8. |
| Orientation overhead was repeated per prompt. | Estimated token reduction: 30-50% on orientation overhead per prompt. |

## Graph Freshness

The `graphify-out/GRAPH_REPORT.md` header shows the commit hash the graph was built from. Run `git rev-parse HEAD` and compare the result to the hash in the report header. If they differ, the graph may be stale; refresh before starting implementation work.

To refresh the normalized graph and Graphify outputs:

```powershell
.\scripts\generate-codebase-graph.ps1
```

The wrapper uses the installed `graphify` CLI from the PyPI package `graphifyy` and normalizes the output into `codebase-graph.json`. It also leaves Graphify's native outputs in `graphify-out/graph.json` and `graphify-out/GRAPH_REPORT.md`.

To refresh using Graphify's own update command, run this in the project root. It has no API cost:

```powershell
graphify update .
```

Use the degraded fallback explicitly only when Graphify is unavailable:

```powershell
.\scripts\generate-codebase-graph.ps1 -FallbackOnly
```

## Graphify CLI Reference

These are the static CLI commands available in the installed Graphify CLI from the `graphifyy` package. Patch 5.2.3 will add the server/tool mode; this section documents static CLI commands only.

| Command | Use |
| --- | --- |
| `graphify update .` | Re-analyze the project and refresh the graph. No API cost; fast. |
| `graphify query . "<natural language question>"` | Ask a question about the codebase structure. |
| `graphify path . <fileA> <fileB>` | Show the dependency/import path between two files. |
| `graphify explain . <symbol>` | Explain what a symbol does and where it is used. |

## Prompt Examples

Example 1 - Routine single-file edit (minimal graph read):

```text
READ FIRST:
1. STATE.json
2. codebase-graph.json    <- find which file owns the relevant symbol
3. src/lib/[target-file]  <- targeted source read, identified from graph
```

Example 2 - Orientation-heavy multi-file feature (full graph read):

```text
READ FIRST:
1. STATE.json
2. codebase-graph.json          <- initial file map
3. graphify-out/GRAPH_REPORT.md <- community structure, god nodes, cross-file relationships
4. src/[fileA], src/[fileB]     <- 2-3 source files identified from graph, not from guessing
```

## Protected Paths

The generator excludes `content/`, `exports/`, `archives/`, `assets/avatars/`, `vault/`, `.env`, `.git`, `.next`, `graphify-out/`, and `node_modules/`. The graph describes engine structure, not creative source material or generated output.
